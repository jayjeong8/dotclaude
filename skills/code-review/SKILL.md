# 코드 리뷰

## 실행 절차

1. `git diff --staged`로 스테이징된 변경사항 확인 (없으면 `git diff`로 전체 변경사항 확인)
2. 수정된 파일에 집중하여 아래 체크리스트 기반으로 리뷰
3. 문제 발견 시 구체적인 수정 예시를 포함하여 피드백

## 체크리스트

### 코드 품질
- 코드가 간결하고 읽기 쉬운가
- 함수와 변수 이름이 적절한가
- 중복 코드가 없는가
- 적절한 에러 처리가 되어 있는가

### 보안
- 노출된 비밀키나 API 키가 없는가
- 입력값 검증이 구현되어 있는가

### 테스트 품질 (필수 검토)

**참조:** `~/.claude/rules/testing-rules.md`

diff에 `.test.ts` / `.test.tsx` / `.spec.ts` 파일이 포함되어 있으면 아래 항목을 **반드시** 점검:

- [ ] **구현 미러링 아닌가?** — "구현 코드를 지우고 다른 구현으로 대체해도 이 테스트가 실패하는가?" 자문. 함수 본체가 한 줄(`=== "value"`)인데 모든 입력을 전수 assert하면 구현 미러링 → **심각** 플래그
- [ ] **False confidence 아닌가?** — mock/ref 연결이 실패하면 초기값만 assert하게 되는 테스트인지 확인. assertion이 실제 로직 경유를 보장하는가?
- [ ] **컴포넌트 렌더링 테스트가 유일한 커버리지인가?** — 조건 분기/계산 로직이 컴포넌트 안에 있고, 테스트는 `render` + `getByText`만 있다면 → 로직을 순수 함수로 **분리 권장**
- [ ] **YAGNI 확장 테스트 아닌가?** — 현재 호출부에서 쓰이지 않는 옵션·파라미터를 테스트하고 있다면 → 제거 권장
- [ ] **vi.mock/jest.mock으로 구현 세부를 스텁해서 실질적 검증이 없진 않은가?**

### 성능
- 성능 고려사항이 반영되어 있는가

## 피드백 우선순위
- 심각 (반드시 수정)
- 경고 (수정 권장)
- 제안 (개선 고려)
