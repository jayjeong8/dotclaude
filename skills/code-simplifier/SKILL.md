# 코드 단순화

기능을 그대로 유지하면서 코드의 명확성, 일관성, 유지보수성을 개선합니다. 가독성 높은 명시적 코드를 과도하게 압축된 코드보다 우선합니다.

## 실행 절차

1. `git diff --staged`로 스테이징된 변경사항 확인 (없으면 `git diff`로 전체 변경사항 확인)
2. 최근 수정된 코드 섹션 식별
3. 아래 원칙에 따라 단순화 기회 분석
4. 프로젝트 코딩 표준(CLAUDE.md) 적용
5. 기능이 변경되지 않았는지 확인
6. 이해에 영향을 주는 주요 변경사항만 문서화

## 단순화 원칙

### 1. 기능 보존 (최우선)
- 코드가 **무엇을 하는지**는 절대 변경하지 않음 - **어떻게 하는지**만 개선
- 모든 원래 기능, 출력, 동작이 그대로 유지되어야 함

### 2. 프로젝트 표준 적용
- CLAUDE.md의 코딩 표준을 따름:
  - ES 모듈과 올바른 import 정렬 및 확장자 사용
  - 화살표 함수보다 `function` 키워드 선호
  - 최상위 함수에 명시적 반환 타입 어노테이션
  - React 컴포넌트에 명시적 Props 타입
  - 적절한 에러 처리 패턴 (가능하면 try/catch 회피)
  - 일관된 네이밍 컨벤션

### 3. 명확성 향상
- 불필요한 복잡성과 중첩 줄이기
- 중복 코드와 불필요한 추상화 제거
- 명확한 변수/함수 이름으로 가독성 개선
- 관련 로직 통합
- 자명한 코드를 설명하는 불필요한 주석 제거
- **중첩 삼항 연산자 금지** - 다중 조건에는 switch 또는 if/else 사용
- 간결함보다 명확함 우선 - 명시적 코드가 압축된 코드보다 나음

### 4. 균형 유지 (과도한 단순화 방지)
다음을 초래하는 과도한 단순화는 피함:
- 코드 명확성이나 유지보수성 감소
- 이해하기 어려운 과도하게 영리한 솔루션
- 너무 많은 관심사를 하나의 함수/컴포넌트에 결합
- 코드 구조를 개선하는 유용한 추상화 제거
- 가독성보다 "줄 수 줄이기" 우선 (중첩 삼항, 밀집된 한 줄 코드 등)
- 디버깅이나 확장이 어려워지는 변경

### 5. 범위 제한
- 최근 수정되거나 현재 세션에서 다뤄진 코드만 개선
- 더 넓은 범위의 검토는 명시적으로 지시받은 경우에만

## 피드백 형식

각 단순화에 대해 다음 형식으로 보고합니다:

```markdown
### [변경 유형] 간단한 설명

**변경 전**
- 해당 코드 스니펫

**변경 후**
- 개선된 코드 스니펫

**이유**: 왜 이 변경이 코드를 더 명확하게 만드는지
```

## 주의사항

- 기능 변경 없이 구조만 개선
- 과도한 추상화나 영리한 코드 지양
- 프로젝트의 기존 패턴과 스타일 존중
- 코드 작성/수정 후 자율적으로 실행 가능
