# Workflow Rules

## 원자적 커밋 (최우선 규칙)

멀티 스텝 계획을 실행할 때, 반드시 각 단계마다 별도로 커밋한다. 여러 변경을 하나의 커밋으로 묶지 않는다. 계획에 번호가 매겨진 단계가 있으면, **각 단계 = 하나의 원자적 커밋**이다.

- **필수 루프**: 한 단계 구현 → 린트 → `/commit` → 다음 단계. 이 루프를 건너뛰고 여러 단계를 연속 구현하는 것은 금지
- TaskUpdate로 단계를 completed 처리하기 전에 반드시 해당 단계의 커밋이 완료되어 있어야 함
- `--amend`는 사용자가 명시적으로 요청할 때만 사용. 기본은 항상 새 커밋 생성

## 린트 → 테스트 → 커밋

코드 수정 후 커밋 전에 항상 수정된 파일에 대해 린트를 실행한다. 기존 테스트가 깨지지 않는지 확인한다. 린트 에러나 테스트 실패가 있으면 커밋 전에 수정한다.

## 큰 작업 분해

복잡한 작업은 먼저 하위 단계로 분해한 후 단계별로 실행한다. 각 단계가 단독으로 해결 가능할 때까지 쪼갠다.

## 컨텍스트 관리

대화가 길어져 성능 저하가 감지되면 `/compact` 사용을 제안한다.

## Git & 커밋

- 커밋 메시지는 별도 지시가 없으면 한국어로 작성
- 커밋 시 `/commit` 스킬 사용
- 사용자가 명시적으로 요청하지 않으면 git push 하지 않기
