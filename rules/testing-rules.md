# Testing Rules (전역)

CLAUDE.md의 필수 테스트 규칙과 함께 모든 TypeScript/JavaScript 프로젝트에 적용된다.

## 테스트 제외 대상
- 단순 props 전달 래퍼 컴포넌트, 페이지 레이아웃, 정적 상수, CSS/애니메이션
- 서드파티 라이브러리 동작 (wretch, zustand, react-query 자체 동작) — 분기·변환·계산·에러 처리 없이 wretch/fetch 체이닝만 하는 API 래퍼 함수도 이에 해당
- barrel export 파일, 생성된 타입 파일
- 조건부 렌더링만 하는 컴포넌트 (로직은 순수 함수/훅으로 분리하여 테스트)
- 단순 비교(`===`) 한 줄짜리 함수 (호출부에서 간접 검증으로 충분)
- DOM ref·렌더 사이클·브라우저 API에 깊이 의존하는 훅 중, jsdom/모킹으로 핵심 동작을 합리적으로 검증하기 어려운 경우 (예: IntersectionObserver·ResizeObserver·스크롤/포커스 위치 추적 등). 단, renderHook + 적절한 모킹으로 검증 가능한 훅은 제외 대상이 아님

## 테스트 용이한 코드 구조
- **API 함수 분리**: API 호출은 `apis/` 또는 유사한 디렉토리에 순수 async 함수로 분리하고, 훅에서는 호출만 한다
- **데이터 변환 순수 함수화**: API 응답 → UI 모델 변환은 훅/컴포넌트 밖의 순수 함수로 추출한다
- **비즈니스 로직 분리**: 계산, 필터링, 상태 판별 로직은 순수 함수로 분리한다
- **훅 의존성 최소화**: 커스텀 훅은 외부 의존을 최소화하고, 계산 로직은 useMemo 내 순수 표현식으로 작성한다
- **컴포넌트 로직 추출**: 컴포넌트 내부의 조건 분기·계산·상태 판별·렌더링 조건 로직은 별도 순수 함수로 추출한다 (컴포넌트 렌더링 테스트 대신 추출된 순수 함수를 테스트)

## 테스트 작성 컨벤션
- 테스트 파일은 소스 파일과 1:1 매칭하여 옆에 colocate (`foo.ts` → `foo.test.ts`)
- 한 소스 파일에 여러 함수가 있으면 하나의 테스트 파일에서 `describe("함수명")` 블록으로 분리
- `describe`는 함수/컴포넌트명(영어), `it`은 한국어 설명
- 테이블 기반 테스트는 `it.each` 활용
- 외부 의존성은 `vi.mock` / `jest.mock`으로 모킹
- React Query 훅 테스트 시 `retry: false`인 테스트용 QueryClient 사용
- `getByRole`/`getByText`를 `getByTestId`보다 우선
- 스냅샷 테스트는 최소화 (대형 컴포넌트 스냅샷 금지)
- **False confidence 테스트 금지**: 테스트 작성 후 반드시 자기 검증 — "이 테스트의 assertion이 실제 로직을 경유하는가?" 확인. mock/ref 연결 실패로 로직에 도달하지 못한 채 초기값·기본값만 assert하는 테스트는 작성하지 않는다
- **구현 미러링 테스트 금지**: "구현 코드를 지우고 다른 구현으로 대체해도 이 테스트가 실패하는가?" 자문. 함수 본체가 한 줄(`=== "value"`)인데 모든 입력을 전수 테스트하는 것은 구현 미러링이다
- **테스트 불가능한 로직은 순수 함수로 분리**: DOM/ref/타이밍에 의존하는 훅 내부 계산 로직은 순수 함수로 추출하여 그 함수를 테스트한다
- **사용처 없는 확장을 테스트하지 않는다**: 현재 호출부가 없는 옵션·파라미터를 추가(YAGNI)하고 그것의 테스트를 작성하는 것은 금지. 추출한 함수의 시그니처는 실제 사용처의 요구에 맞춘다

## 구현-테스트 워크플로우
- 구현 코드와 해당 테스트는 **같은 커밋**에 포함한다 (구현만 커밋 후 테스트를 나중에 작성하지 않는다)
- 원자적 커밋 루프: 기능 구현 + 테스트 작성 → 린트 → 테스트 실행 → `/commit` → 다음 단계
- 기존 코드 수정 시: 기존 테스트를 먼저 실행하여 현재 상태를 확인한 후 수정에 착수한다
